# Асыл-LTD — CRM для учёта сырья и склада (MVP) — Дизайн

**Дата:** 2026-06-18
**Статус:** согласован, готов к написанию плана реализации

## 1. Контекст и цель

«Асыл-LTD» — мукомольный цех (пшеничная мука Премиум, до 90% на экспорт:
Узбекистан, Афганистан, Кыргызстан, Китай, Иран и др.). Системного учёта в
цехе сейчас **нет**. Цель — построить с нуля внутреннюю CRM для учёта сырья,
склада готовой муки, заказов и отгрузок.

В будущем планируется автоматизированный пост отгрузки: умные камеры (чтение
номера машины, подсчёт мешков), весы и автоматические ворота. **В рамках этого
MVP железо НЕ подключается** — все «автоматические» данные вводятся вручную
оператором, но хранятся в тех же полях/таблицах, куда позже будет писать
железо. Это ключевой принцип: переход на железо не должен требовать переделки
моделей данных.

### Целевой флоу отгрузки (что автоматизируем в будущем, что моделируем сейчас)

1. КамАЗ приезжает, встаёт на весы.
2. Камера считывает номер машины *(MVP: оператор вводит вручную)*.
3. Система проверяет: есть ли по номеру заказ и оплачен ли он.
4. Если оплачен → ворота открываются *(MVP: оператора пускают)*, статус заказа
   → `arrived`, в журнал пишется время прибытия и **вес въезда**.
5. Идёт загрузка, камера считает мешки *(MVP: оператор вводит кол-во мешков)* →
   запись в журнал.
6. КамАЗ выезжает, снова встаёт на весы → **вес выезда**.
7. Система считает `вес муки = |вес выезда − вес въезда|` и сверяет с расчётом
   по мешкам; результат пишется в журнал.

## 2. Архитектура и стек

Два отдельных приложения:

- **Бэкенд:** Django + Django REST Framework (DRF) + PostgreSQL. Отдаёт REST API
  (JSON). Django-админка остаётся для суперпользователя (служебно) — и через неё
  ведутся справочники (сорта, фасовки).
- **Фронт:** Next.js (React) — отдельный проект, общается с API по HTTP.
  Обслуживает **две аудитории**:
  - **внутренний интерфейс** для сотрудников цеха (менеджер, бухгалтер,
    оператор, начальник);
  - **клиентский портал** — отдельный кабинет, где клиент сам создаёт и
    отслеживает свои заказы.
- **Аутентификация:** JWT (`djangorestframework-simplejwt`), access + refresh
  токены. Пользователи делятся на два типа: **сотрудники** (роли из §5) и
  **клиенты** (доступ только к своим данным через портал).

### Готовность к железу

Камеры/весы/ворота позже будут слать данные в те же API-эндпоинты
(`arrive` / `load` / `ship`) и/или писать в `EventLog`, что и ручной ввод
сейчас. Модели данных при этом не меняются — меняется лишь источник данных.

## 3. Модель данных

- **Client** — клиент: название, контакты (обязательны); страна и реквизиты
  (ИНН и т.п.) — **опциональные**. Может быть связан с учётной записью
  пользователя-клиента для доступа в портал.
- **Grade** — справочник сортов муки (Премиум и др.). Управляется из админки:
  можно добавлять/убирать значения. Название, активность.
- **Packaging** — справочник фасовок (50/25/10 кг). Управляется из админки.
  Название, вес в кг, активность.
- **Product** — номенклатура муки = комбинация **Grade × Packaging** (например
  «Премиум 50 кг») + цена. Создаётся из существующих значений справочников, не
  зашит в код.
- **StockItem** — остаток по каждому Product **в мешках**. Меняется при приёмке
  и отгрузке. Вес — расчётный (кол-во мешков × вес фасовки).
- **StockReceipt** — приёмка муки на склад: Product, кол-во мешков, дата, кто
  принял. Увеличивает остаток.
- **Order** — заказ: клиент, позиции, сумма, статус, номер машины (КамАЗ).
- **OrderItem** — позиция заказа: Product + количество мешков.
- **Payment** — оплата по заказу: сумма, дата, кто отметил. Заказ считается
  оплаченным, когда сумма оплат ≥ суммы заказа.
- **Shipment** — факт отгрузки по заказу: вес въезда (тара), вес выезда,
  расчётный вес муки, кол-во загруженных мешков, время. Флаг и автор разрешения
  отгрузки в долг (если применялось).
- **EventLog** — журнал событий (неизменяемый): «машина прибыла», «начата
  загрузка», «загружено N мешков», «выезд, вес X», «расхождение по весу»,
  «отгрузка в долг разрешена …». Время, пользователь, тип события, данные.
  Сюда позже пишет железо.

## 4. Статусы заказа и жизненный цикл

| Статус | Что значит | Кто/что меняет |
|---|---|---|
| `draft` | Заказ создан, черновик | Менеджер или **клиент** (через портал) |
| `confirmed` | Подтверждён, ждёт оплаты | Менеджер |
| `paid` | Оплачен полностью | Авто: сумма оплат ≥ суммы заказа |
| `arrived` | Машина прибыла, проверена, въехала (вес въезда) | Оператор |
| `loading` | Идёт загрузка, считаются мешки | Оператор |
| `shipped` | Выезд, вес выезда, расчёт веса, склад списан | Оператор |
| `cancelled` | Отменён | Менеджер |

### Бизнес-правила переходов

1. Переход в `arrived` **запрещён**, если заказ не `paid` — это правило «ворота
   открываются только если оплачено». **Исключение:** пользователь с ролью
   **Начальник** может разрешить отгрузку в долг — тогда переход разрешён, но в
   `EventLog` пишется кто разрешил и когда (жёсткое по умолчанию + разрешение
   начальника).
2. При переходе в `shipped` (в одной транзакции БД):
   - считается `вес муки = |вес выезда − вес въезда|`;
   - со склада **списываются мешки** по позициям заказа;
   - в журнал пишется расхождение: расчёт по мешкам (кол-во × вес фасовки)
     против факта по весам.
3. Каждый переход статуса добавляет запись в `EventLog`.

Склад списывается в момент `shipped` (по факту вывоза), а не при создании
заказа.

## 5. Роли и права

Через группы Django + проверки прав в DRF. Один человек может иметь несколько
ролей.

| Роль | Что может |
|---|---|
| **Менеджер** | Клиенты, товары, создание/редактирование заказов, статусы до `paid` |
| **Бухгалтер** | Отмечает оплаты, видит долги/отчёты по оплатам |
| **Оператор отгрузки** | Прибытие, веса, мешки; переходы `arrived → loading → shipped` |
| **Начальник** | Всё выше + разрешение отгрузки в долг, все отчёты |
| **Админ (суперпользователь)** | Полный доступ, Django-админка, управление пользователями и справочниками |

Каждый чувствительный переход статуса проверяет роль на сервере.

### Клиент (внешний пользователь)

Отдельный тип учётной записи, **не** входит в роли сотрудников. Клиент через
портал: видит каталог товаров и цены, создаёт свои заказы, видит статусы и
оплаты **только по своим заказам**. Не имеет доступа к чужим данным, складу,
журналу и внутренним экранам. Оплату клиент совершает вне системы; подтверждает
её бухгалтер.

## 6. API (DRF)

**Внутренние (сотрудники):**

- `POST /api/auth/login`, `POST /api/auth/refresh` — JWT.
- `/api/clients/` — CRUD клиентов.
- `/api/grades/`, `/api/packagings/` — справочники сортов и фасовок (чтение для
  фронта; ведение — в админке).
- `/api/products/` — CRUD товаров (Grade × Packaging + цена).
- `/api/stock/` — остатки (чтение).
- `/api/stock/receipts/` — приёмка муки (+мешки).
- `/api/orders/` — CRUD заказов с позициями.
- `/api/orders/{id}/payments/` — добавить оплату (бухгалтер).
- `/api/orders/{id}/arrive/` — фиксация прибытия (номер машины, вес въезда);
  проверка оплаты / разрешения начальника.
- `/api/orders/{id}/load/` — фиксация загрузки (кол-во мешков).
- `/api/orders/{id}/ship/` — фиксация выезда (вес выезда) → расчёт веса,
  списание склада.
- `/api/events/` — журнал событий (чтение, фильтры).
- `/api/reports/...` — сводки (приёмка / отгрузка / остатки / долги).

**Портал клиента (доступ только к своим данным):**

- `POST /api/portal/auth/login`, `refresh` — вход клиента (JWT).
- `/api/portal/catalog/` — каталог товаров с ценами (чтение).
- `/api/portal/orders/` — создание и просмотр **своих** заказов с позициями.
- `/api/portal/orders/{id}/` — статус, позиции, оплаты по своему заказу.

На сервере все портальные эндпоинты фильтруют данные по клиенту из токена —
клиент не может увидеть чужой заказ даже по прямому ID.

## 7. Экраны фронта (Next.js)

1. **Логин**
2. **Дашборд** — машины на территории, заказы к отгрузке, остатки, долги.
3. **Клиенты** — список + карточка.
4. **Товары и склад** — остатки в мешках, приёмка муки.
5. **Заказы** — список, карточка заказа (позиции, оплаты, статус, переходы).
6. **Пост отгрузки** — отдельный рабочий экран оператора: очередь машин, ввод
   номера машины, веса въезда/выезда, мешков; кнопки `arrive / load / ship`.
   Именно сюда позже подключается железо.
7. **Журнал** — лента событий с фильтрами.
8. **Отчёты** — приёмка, отгрузка, остатки, долги по клиентам.

### Кабинет клиента (портал)

Отдельный раздел фронта со своим входом:

- **Логин клиента**
- **Каталог** — товары и цены.
- **Новый заказ** — выбор товаров, количества мешков; отправка заказа.
- **Мои заказы** — список своих заказов, статусы, оплаты, реквизиты для оплаты.

## 8. Целостность, ошибки, тестирование

### Целостность данных

- Списание склада при `ship` — в **транзакции БД**: либо всё (расчёт веса +
  списание мешков + запись лога), либо ничего.
- Нельзя списать больше мешков, чем есть на складе → понятная ошибка.
- Запрет «двойной отгрузки»: повторный `ship` по отгруженному заказу отклоняется.
- Все переходы статусов проверяются на сервере (фронту не доверяем): неверный
  переход (например `draft → shipped`) → ошибка 400.

### Журнал

`EventLog` неизменяем — записи только добавляются, не редактируются и не
удаляются. Источник правды по событиям.

### Обработка ошибок API

Единый формат JSON-ошибок (`{"detail": "...", "code": "..."}`), внятные
сообщения на русском для фронта.

### Тестирование (TDD)

- Юнит-тесты на бизнес-правила: переходы статусов; правило «нет оплаты → нет
  въезда (кроме разрешения начальника)»; расчёт веса муки; списание склада.
- Тесты на права ролей (оператор не может отметить оплату и т.п.).
- Тесты изоляции портала: клиент видит и меняет **только свои** заказы, не
  имеет доступа к складу/журналу/чужим данным.
- Сначала тест на правило, потом реализация.

## Вне рамок MVP (YAGNI)

- Подключение камер, весов, ворот (только подготовка моделей/API).
- Интеграция с банком/эквайрингом (оплата отмечается вручную).
- Учёт производства/помола, смен и зарплат сотрудников.
- Мобильное приложение.
