# Камеры и вебхуки отгрузки — Дизайн

**Дата:** 2026-06-22
**Статус:** согласован, готов к написанию плана реализации

## Цель

Автоматизировать пост отгрузки через камеры. Камера считывает госномер машины
(а камера-счётчик — ещё и количество мешков) и шлёт данные на наш сервер по
вебхуку. Сервер находит заказ по номеру, выполняет нужное действие (тот же
сервис, что и ручной «Пост отгрузки») и **отвечает камере по настраиваемому
JSON-шаблону** — чтобы подойти под разные модели камер. Каждый вызов
логируется.

Нодовый визуальный редактор правил — **вне рамок этой итерации** (отдельный
проект; модель «камера → действие → шаблон ответа» закладывается так, чтобы
редактор позже стал лишь другим UI над теми же данными).

## 1. Модель данных

Новое приложение **`webhooks`**.

- **Camera**
  - `name` — человекочитаемое имя.
  - `camera_id` — уникальный код (напр. `gate-01`, `counter-01`, `exit-01`).
  - `kind` — тип: `entry` (въезд) / `counter` (счётчик загрузки) / `exit` (выезд).
  - `api_key` — секрет, генерируется криптостойко.
  - `response_template` — JSON-шаблон ответа с `{{переменными}}` (может быть пуст
    → дефолтный формат).
  - `is_active` — активна ли.
  - `last_seen` — время последнего вызова (null до первого).
- **WebhookCall** (журнал вызовов)
  - `camera` FK, `plate` (нормализованный номер), `payload_bags` (int null),
    `payload_weight` (Decimal null), `matched_order` FK→Order null,
    `decision` (`allow`/`deny`), `reason` (текст), `request_payload` (JSON),
    `response_payload` (JSON), `created_at`.

**Связь с заказом:** поиск по `Order.truck_number` с нормализацией номера
(убрать пробелы, верхний регистр).

## 2. Эндпоинт вебхука

`POST /api/webhook/camera/` — без JWT, своя аутентификация по ключу. Исключён
из общих DRF-permissions.

**Тело запроса:**
```json
{ "camera_id": "gate-01", "plate": "123ABC02", "bags": 50, "weight_kg": 10500 }
```
- `camera_id` — обязателен (по нему ищется камера).
- `plate` — обязателен.
- `bags` — для `counter`. `weight_kg` — для `exit`. Лишние поля игнорируются.

**Заголовок:** `X-Camera-Key: <api_key>`.

**Идентификация и проверки (по порядку):**
1. `camera_id` из тела → найти камеру. Нет → `404`.
2. Сверить `X-Camera-Key` с `camera.api_key`. Не совпал → `401`.
3. Камера неактивна → `403`.
4. Нормализовать `plate` → найти заказ по `truck_number`.
5. По `kind` вызвать сервис отгрузки (см. §1). Сервис сам проверяет статус и
   правила (оплата, склад и т.д.).
6. Успех → `decision=allow`; бизнес-ошибка (нет заказа / не оплачен / неверный
   статус) → `decision=deny` + причина.
7. Записать `WebhookCall`, обновить `camera.last_seen` (в транзакции).
8. Ответить по шаблону камеры (§3) с HTTP `200`.

**Действия по типу** (переиспользуют существующие сервисы, без дублирования):
- `entry`: `record_arrival` → заказ `arrived`.
- `counter`: `record_loading(bags)` → заказ `loading`, мешки записаны.
- `exit`: `record_shipment(weight_kg)` → заказ `shipped`, нетто/списание склада.

**HTTP-коды:** `200` с телом (allow/deny внутри) для бизнес-логики; `401/403/404`
— только для ключа/камеры.

## 3. Шаблон ответа с переменными

В админке у камеры — JSON-шаблон с плейсхолдерами `{{переменная}}`. Сервер
делает **безопасную подстановку** (без eval) и валидирует, что получился
корректный JSON.

**Переменные:** `{{camera_id}}`, `{{decision}}` (`"allow"`/`"deny"`),
`{{allowed}}` (`true`/`false`), `{{reason}}`, `{{order_id}}`, `{{plate}}`,
`{{client_name}}`, `{{bags}}`, `{{weight_kg}}`, `{{net_weight_kg}}`.

Булевы/числа подставляются без кавычек; строки экранируются. Пустой шаблон →
дефолт: `{"decision": "...", "allowed": ..., "order_id": ..., "reason": "..."}`.

**Пример** (камера въезда):
```json
{ "open_gate": {{allowed}}, "order": {{order_id}}, "message": "{{reason}}" }
```
При разрешении вернётся `{ "open_gate": true, "order": 42, "message": "" }`.

## 4. UI камер

Раздел **«Камеры»** в сайдбаре (под «Управление»), права `cameras.view` /
`cameras.manage`.

- **Список камер** + создание (модалка): имя, `camera_id`, тип, шаблон ответа
  (textarea JSON), активность. Ключ генерируется при создании, показывается
  (копировать), можно перегенерировать. В строке: статус (онлайн по
  `last_seen`), тип, последний вызов.
- **Карточка камеры**: данные + ключ (маскирован, раскрыть/перегенерировать);
  редактор шаблона с подсказкой переменных; **«Симулировать вызов»** (ввести
  номер + мешки/вес → увидеть ответ сервера, без реальной камеры); **журнал
  вызовов этой камеры** (время, номер, результат, заказ, причина).
- **Журнал вебхуков** — все вызовы всех камер с фильтрами.

## 5. Права, целостность, ошибки, тестирование

### Права (RBAC)
Добавить раздел `cameras` в каталог `rbac/perms.py`: `cameras.view`,
`cameras.manage`. Управление/симулятор — `cameras.manage`; просмотр/журнал —
`cameras.view`. Пресет «Начальник» и админ получают их.

### Целостность
- Обработка вызова — в **транзакции** (сервис + WebhookCall атомарно).
- `camera_id` уникален; `api_key` криптостойкий, в списках маскируется (полный —
  только при создании/перегенерации).
- Вебхук вне JWT; аутентификация только по ключу.

### Ошибки
- Ключ/камера → `401/403/404`.
- Бизнес-ошибки → `200` + `decision: "deny"` + причина.
- Невалидный шаблон при сохранении → `400` (проверка: после подстановки —
  валидный JSON). Формат ошибок `{"detail","code"}`, русский текст.

### Тестирование (TDD)
- Идентификация: верный ключ → ок; неверный → 401; нет `camera_id` → 404;
  неактивна → 403.
- entry: оплаченный заказ → allow + `arrived`; не оплачен → deny.
- counter: заказ `arrived` + bags → `loading` + мешки записаны.
- exit: заказ `loading` + weight → `shipped` + нетто/списание.
- Шаблон: подстановка даёт валидный JSON; булевы/числа без кавычек; пустой
  шаблон → дефолт.
- `WebhookCall` пишется на каждый вызов; `last_seen` обновляется.
- Симулятор возвращает тот же ответ, что реальный вызов, без побочных эффектов
  на заказ (или с пометкой — см. план).

## Вне рамок (YAGNI)

- Нодовый визуальный редактор правил (отдельный проект).
- Инкрементальный счёт мешков (по одному) — шлём итог.
- Подпись запроса (HMAC) сверх API-ключа.
- Реальная интеграция с воротами (сервер лишь отвечает; железо реагирует само).
