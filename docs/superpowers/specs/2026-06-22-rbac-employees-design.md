# Динамические роли и сотрудники (RBAC) — Дизайн

**Дата:** 2026-06-22
**Статус:** согласован, готов к написанию плана реализации

## Цель

Заменить захардкоженные роли (`IsManager`/`IsAccountant`/`IsOperator`/`IsBoss`)
на **динамическую систему прав**: админ создаёт учётные записи сотрудников,
назначает каждому **одну роль**, а роли создаются и настраиваются динамически —
права назначаются **по разделам и действиям** (view/create/edit/… на раздел).

**Главный приоритет — чистота:** единый каталог прав в коде, один
permission-класс, проверка прав в одном месте, RESTful API, один фронт-хелпер
`can()`. Никаких `if user.is_manager` по коду.

## 1. Модель данных

Новое приложение **`rbac`** + расширение `accounts`.

- **Permission** — справочник прав вида `<раздел>.<действие>`:
  `code` (уникальный), `section`, `action`, `label`. Заполняется
  data-миграцией из единого каталога в коде (`rbac/perms.py`). Пользователь
  новые права НЕ создаёт — только комбинирует существующие.
- **Role** — динамическая роль: `name`, `description`, `is_system` (для
  пресетов), M2M `permissions` → Permission. Создаётся/редактируется админом.
- **Employee** — профиль сотрудника, `OneToOne` на `accounts.User`:
  `first_name`, `last_name`, `phone`, `position` (должность),
  `role` (FK → Role, одна роль), `is_active`. `User` хранит логин/пароль;
  Employee — профиль и роль.

**Цепочка прав:** `User → Employee → Role → Permissions`.

Метод на User: `has_perm_code(code) -> bool` — `True` для суперюзера; иначе
проверяет, что `user.employee.role.permissions` содержит код. Свойство
`perm_codes` — множество кодов прав пользователя (для `/auth/me/`).

## 2. Каталог прав (единый источник истины)

`rbac/perms.py` — единственное место определения кодов. Все ссылки (permission-
классы, сидинг, тесты) импортируют отсюда.

| Раздел | Действия |
|---|---|
| `catalog` (Номенклатура) | view, create, edit, delete |
| `clients` | view, create, edit, delete |
| `warehouse` (Склад) | view, adjust |
| `orders` | view, create, edit, confirm |
| `payments` | view, create |
| `shipping` (Пост отгрузки) | view, arrive, load, ship, debt_override |
| `events` (Журнал) | view |
| `reports` (Отчёты) | view |
| `employees` (Сотрудники/роли) | view, manage |

### Системные пресеты (создаются data-миграцией, `is_system=True`)

- **Менеджер** — catalog.*, clients.*, orders.*, payments.view, reports.view, events.view.
- **Бухгалтер** — orders.view, payments.view/create, reports.view, events.view, clients.view.
- **Оператор** — shipping.view/arrive/load/ship, orders.view, warehouse.view, events.view.
- **Начальник** — всё вышеперечисленное + shipping.debt_override + reports.view.

Пресеты можно редактировать (менять права), но нельзя удалить.

## 3. Проверка прав на бэкенде

Один класс `rbac.permissions.HasPerm`. Во вьюхах — атрибут
`required_perms: dict[action, code]`; `get_permissions()` берёт код по текущему
`self.action` и возвращает `HasPerm(code)`. Суперюзер проходит всегда.

Старые классы (`IsManager`/`IsAccountant`/`IsOperator`/`IsBoss`/
`IsOperatorOrBoss`) **удаляются**. Везде, где они использовались
(catalog/clients/orders/warehouse/shipments/eventlog views), переходим на
`required_perms`. `IsStaff`/`IsClientUser` остаются (отличают сотрудника от
портального клиента).

`record_arrival`: правило долга проверяет `user.has_perm_code("shipping.debt_override")`
вместо `user.is_boss`.

## 4. API

Новые эндпоинты (приложения `rbac`, `employees`):

- `GET /api/permissions/` — все права (read-only), для редактора роли.
- `/api/roles/` — CRUD ролей (ViewSet). Назначение прав — список кодов/ID.
  Системные роли: редактируются, не удаляются. Право `employees.manage`.
- `/api/employees/` — CRUD сотрудников (ViewSet). Создание (транзакция):
  создаёт `User` (username, password) + `Employee` (ИО, телефон, должность,
  роль). Право `employees.manage` на запись, `employees.view` на чтение.
- `GET /api/auth/me/` — расширяется полем `permissions: [...]` (коды прав
  текущего пользователя) для фронт-гейтинга.

Пароль — только на запись (`write_only`), через `set_password`, в ответах
никогда не возвращается.

## 5. Фронт

- Хелпер `can(code)` (читает `me.permissions`; суперюзер → всегда `true`) +
  использование везде единообразно.
- **Навигация по правам:** пункт меню виден при наличии `<раздел>.view`
  (вместо захардкоженных ролей в `sidebar.tsx`).
- Кнопки/действия (создать, удалить, отгрузить…) гейтятся через `can()`.
- Новый раздел сайдбара **«Управление»** (виден при `employees.view`):
  - **Сотрудники** — таблица + модалка создания (логин, пароль, ИО, телефон,
    должность, роль).
  - **Роли** — список ролей + редактор: матрица галочек «раздел × действие».

## 6. Целостность, ошибки, тестирование

### Целостность
- Создание сотрудника — в **транзакции** (User + Employee атомарно).
- Удаление роли запрещено, если на неё назначены сотрудники → понятная ошибка
  (не каскад).
- Системные пресеты неудаляемы (`is_system`), права править можно.
- Пароль — `set_password` (хеш), никогда не хранится/не отдаётся в открытом виде.

### Ошибки
Единый формат `{"detail","code"}`, сообщения на русском. Нет права → `403`.

### Тестирование (TDD)
- Каталог прав консистентен (коды уникальны; сид создаёт все коды из enum).
- `HasPerm`: пускает при наличии права, отклоняет без него, суперюзер всегда.
- Создание сотрудника транзакционно; нельзя удалить роль с сотрудниками;
  пресеты неудаляемы.
- `/auth/me/` отдаёт корректный набор `permissions` для роли.
- `shipping.debt_override` управляет отгрузкой в долг.

## Вне рамок (YAGNI)

- Несколько ролей на сотрудника / индивидуальные правки прав поверх роли.
- Сброс/смена пароля сотрудником, восстановление по email.
- Аудит изменений ролей (кроме общего EventLog, если потребуется позже).

## Миграция существующего

Группы Django (`manager` и т.д.) и сидинг групп удаляются/заменяются на роли
`rbac`. Существующий суперпользователь (admin) не затрагивается — проходит
всегда. Логика `is_manager`/`is_boss` на модели User удаляется.
