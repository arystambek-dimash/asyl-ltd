# Архивирование товаров (и статус архива заказов)

## Контекст

Пользователь попросил «добавить архивирование заказов и товаров». В диалоге выяснилось: **корзина и архив — это одно и то же** («убрать с глаз, можно вернуть»).

Текущее состояние:
- **Заказы** уже имеют полноценный архив: soft-delete через `Order.deleted_at`/`deleted_by`, вкладка «Корзина», восстановление, автоматическое исключение из всех отчётов и агрегатов (через `LiveOrderManager` как default-менеджер). Здесь **делать нечего**.
- **Товары** имеют флаг `Product.is_active` (bool) с кнопкой-тумблером «Скрыть/Включить» и бейджем «Активен/Скрыт». По смыслу это уже архив, но:
  - нет вкладки «Архив» и явного восстановления как у заказов;
  - `destroy` товара делает hard-delete (который всё равно падает из-за `PROTECT` от `OrderItem`);
  - **латентный баг:** формы заказа (`order-form.tsx`, `city/orders/page.tsx`, `warehouse/page.tsx`) показывают в выпадашке товаров ВСЕ товары, включая скрытые — `/products/` отдаёт всё без фильтра.

## Решения пользователя

1. Товарам — **такую же корзину/архив**, как у заказов (кнопка, вкладка, восстановление).
2. Архивный товар **скрыт только из выбора новых заказов** и прайс-листов; старые заказы и все отчёты работают как обычно, аналитика не меняется.
3. Терминология — **«Архив» везде** на странице товаров (вкладка «Архив», кнопка «В архив»/«Восстановить», бейдж «В архиве»).

## Ключевое архитектурное решение: переиспользуем `is_active`, НЕ плодим `deleted_at`

`Product.is_active` уже выражает «в архиве / не в архиве». Добавлять товару отдельные `deleted_at`/`deleted_by` (как у заказов) — дублирование состояния с двумя источниками правды. Вместо этого:

- `is_active=True` → активный товар (виден в списке «Товары», доступен в новых заказах);
- `is_active=False` → **в архиве** (виден только во вкладке «Архив», недоступен в новых заказах, но остаётся в старых заказах и отчётах).

Это соответствует памяти [[clean-extensible-code]] — без дублирования, логика централизована.

## Backend

### `apps/catalog/views.py` — `ProductViewSet`
- **Дефолтный queryset** (`list`/`retrieve` для форм заказа) отдаёт только активные: `Product.objects.filter(is_active=True)`, кроме случая `?archived=1`.
- Queryparam `?archived=1` → `filter(is_active=False)` (для вкладки «Архив»).
- Queryparam `?all=1` → без фильтра (на случай админских экранов; опционально, только если уже используется — иначе не добавляем, YAGNI).
- `@action(detail=True, methods=["post"], url_path="archive")` → `is_active=False`, право `catalog.edit`. Идемпотентно (если уже в архиве — вернуть без ошибки или 400 с кодом `already_archived`, по образцу `soft_delete_order`).
- `@action(detail=True, methods=["post"], url_path="restore")` → `is_active=True`, право `catalog.edit`.
- `destroy` → **архивирование вместо hard-delete** (ставит `is_active=False`), по образцу `OrderViewSet.destroy`. Право остаётся `catalog.delete`.

Логику архива/восстановления вынести в маленькие сервис-функции `archive_product(product, user)` / `restore_product(product, user)` в `apps/catalog/services.py` (создать файл, если его нет) — по образцу `soft_delete_order`/`restore_order`, с `log_event` («Товар в архив» / «Товар восстановлен»). Это держит логику централизованной и тестируемой.

### `apps/catalog/serializers.py`
- `is_active` уже в fields (read/write) — оставляем.

### Логика фильтрации в других местах
- `ClientPricesView` и любые места, где товар подставляется в НОВЫЙ заказ, должны учитывать только активные. Проверить `apps/orders` на прямые `Product.objects.all()` при создании заказа — если фильтра по `is_active` нет, добавить (но старые `OrderItem` не трогать).

## Frontend

### `frontend/src/app/catalog/products/page.tsx`
- Вкладки **[Товары] [Архив]** (паттерн Tabs как в `orders/page.tsx`: [Заказы][Корзина]).
- «Товары»: активные (`/products/`), кнопка **«В архив»** (с `ConfirmDialog`, как удаление заказа), кнопка «Изменить», редактирование цены.
- «Архив»: `/products/?archived=1`, кнопка **«Восстановить»**, бейдж «В архиве».
- **Убрать** старый тумблер «Скрыть/Включить» и парный бейдж «Активен/Скрыт» — их роль поглощает архив (одна концепция вместо двух). `toggleActive` удаляется.
- Счётчики вкладок: число активных / число в архиве.

### `frontend/src/lib/types.ts`
- `Product.is_active` уже есть — изменений в типе не требуется.

### Формы заказа — фикс латентного бага
После бэкенд-правки `/products/` уже отдаёт только активные, поэтому `order-form.tsx`, `city/orders/page.tsx`, `warehouse/page.tsx` автоматически перестанут показывать архивные товары. **Проверить**, что ни одна из этих форм не запрашивает `/products/?all=1` или не полагается на скрытые товары. Изменений в самих формах, скорее всего, не нужно — только верификация.

## Что НЕ делаем (YAGNI)

- НЕ добавляем товару `deleted_at`/`deleted_by` — `is_active` уже покрывает состояние.
- НЕ трогаем архив заказов — он готов и работает.
- НЕ переименовываем вкладку заказов «Корзина» (пользователь выбрал вариант без переименования заказов).
- НЕ добавляем `?all=1`, если он нигде не нужен.
- НЕ меняем аналитику/отчёты — архивный товар в старых заказах считается как обычно.

## Тесты (backend)

`apps/catalog/tests/` (создать, если нет):
- `list` без параметров → только активные товары.
- `?archived=1` → только архивные.
- `POST /products/{id}/archive/` → `is_active=False`, право `catalog.edit` (без права — 403).
- `POST /products/{id}/restore/` → `is_active=True`.
- `DELETE /products/{id}/` → архивирует (is_active=False), НЕ удаляет из БД; товар остаётся привязан к своим `OrderItem`.
- Архивный товар: старый заказ с ним по-прежнему считается в `total_amount` и отчётах (архив товара не влияет на существующие заказы).
