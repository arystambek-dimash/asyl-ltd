# Переработка флоу заказов и оплат

Дата: 2026-06-24

## Проблема

Сейчас оплата зашита в середину логистической цепочки:
`confirmed → arrived → paid → loading → loaded → shipped`. Загрузка не
начинается, пока клиент не оплатил (`start_loading` требует `status == "paid"`).
Это неверно: ~90% заказов берётся в долг, и груз должен ехать до оплаты.

## Цель

Оплата (или фиксация долга) происходит **после** статуса «Отгружен». Долг —
основной путь. Загрузка идёт без оплаты как блокера. Добавляются магазины с
расписанием оплат.

## Новый флоу

Логистический и платёжный статусы **разнесены**.

**Логистика** (`order.status`):
```
draft → pending → confirmed → arrived → loading → loaded → shipped
                  └ rejected / cancelled
```
- `loading` разрешён сразу после `arrived` (оплата больше не блокирует).
- Статус `paid` из середины цепочки **удаляется**.

**Оплата** (`order.payment_status`): `unpaid → partial → settled`.
- Появляется только после `shipped`: `record_shipment` фиксирует весь заказ как
  долг и ставит `payment_status = unpaid`.
- Оплата возможна **только** когда `status == "shipped"`. Поэтапно (частями) или
  полной суммой. Полное погашение → `settled`.

**Намерение расчёта** (`order.settlement_intent`): `debt` (по умолчанию) /
`instant`. Выбирается клиентом при создании заказа. Это пометка способа расчёта,
а не оплата — реальные деньги/банковский запрос идут после `shipped`.
`instant` → банковский запрос (заглушка пока).

## Уведомление о номере КАМАЗа

После одобрения заказа отправляющий вписывает номер КАМАЗа (`set_truck_number`).
При этом создаётся уведомление клиенту.

Новая лёгкая модель `notifications.Notification`:
- `client` FK, `text`, `is_read` (bool), `created_at`.
Клиент видит уведомления в кабинете. Push/SMS — позже.

## Магазины и расписание оплат

Новая модель `clients.Store`:
- `client` FK (владелец; у клиента может быть несколько магазинов)
- `name`, контакты (`address`/`phone`)
- `payment_schedule_type`: `none` / `monthly` / `weekly`
- `payment_days`: список дней (числа месяца `[5, 20]` или дни недели). Задаёт
  **операционист** при заключении договора.
- `contract_signed_at` (дата, опционально)

`orders.Order` получает `store` FK (nullable) — заказ может быть от магазина или
напрямую от клиента-физлица.

**Окно оплаты — строгое.** Оплата долга по заказу магазина разрешена только если
сегодня входит в `payment_days`. Вне окна — ошибка («Оплата для магазина «X»
доступна только N-го числа»). Централизованная проверка в сервисе оплаты.

**Просрочка.** Если в день оплаты магазин не заплатил, система фиксирует
просрочку и шлёт уведомление. Реализуется сервис-функцией (детектор), вызов по
крону — позже. Пока — функция + ручной/тестовый вызов.

Заказы физлиц (без магазина или `schedule_type=none`) платят в любой день,
поэтапно или полностью.

## Затрагиваемые модели

- `clients.Store` — **новая**
- `orders.Order` — `payment_status`, `settlement_intent`, `store` FK; статус
  `paid` больше не используется как блокер
- `orders.Payment` — без изменений
- `notifications.Notification` — **новая** (новый app или в clients)

## Сервисы (логика централизована, без дублирования)

- `orders/services.py`:
  - новые `ALLOWED_TRANSITIONS`: `arrived → loading` напрямую
  - `record_shipment` создаёт долг, ставит `payment_status = unpaid`
  - `add_payment`: обновляет `payment_status`; проверяет окно оплаты магазина;
    оплата только при `status == "shipped"`
  - `set_truck_number`: создаёт Notification
  - `_maybe_mark_paid` → переводит в `payment_status = settled` (не трогает
    логистический статус)
- `shipments/services.py`: убрать проверку `status == "paid"` в `start_loading`
  и `record_count`
- сервис магазинов (в clients): `is_payment_window_open(store, date)` +
  `detect_overdue_stores()`

## Тестирование (TDD)

- поэтапная оплата → `partial`, полное погашение → `settled`
- загрузка идёт без оплаты
- блокировка оплаты магазина вне окна; разрешение в окне
- детектор просрочки создаёт уведомление
- уведомление о номере КАМАЗа создаётся
