# Редизайн флоу заказа и поста отгрузки

**Дата:** 2026-06-22
**Статус:** утверждён к реализации

## Цель

Сделать флоу поста отгрузки чистым и предсказуемым: КАМАЗ привязывается к заказу
по **номеру** на всех этапах; прибытие — автоматическое (камера/датчик), оператор
только смотрит; загрузка считает мешки CV в реальном времени; после явной кнопки
«Загрузка завершена» открывается этап выезда, где вводится вес выезда и система
**сравнивает** вес груза с расчётом по мешкам.

## Сквозной принцип: привязка по номеру КАМАЗа

Номер КАМАЗа вписывается в заказ **заранее** (менеджером). Все вебхуки камер
(entry / counter / exit) находят заказ по совпадению нормализованного
`truck_number`. Оператор заказ вручную не выбирает. Если номер не найден —
вебхук отвечает `deny` и пишет запись в журнал вебхуков (оператор видит
«нераспознан»). Поведение для МВП — без ручной привязки.

## Флоу статусов

Новый набор статусов заказа:
`draft → confirmed → paid → arrived → loading → loaded → shipped` (+ `cancelled`).

Добавляется один статус — **`loaded`** (между `loading` и `shipped`).

| Переход | Триггер | Кто |
|---|---|---|
| `confirmed/paid → arrived` | вебхук entry (номер + вес въезда) **или** ручной fallback на посту | камера / оператор |
| `arrived → loading` | загрузка тест-видео в CRM (старт обработки) | оператор |
| (счёт мешков) | `+1` в Redis на каждый мешок — статус НЕ меняется | воркер |
| CV `complete` | фиксирует число мешков в shipment, статус остаётся `loading` | воркер |
| `loading → loaded` | кнопка «Загрузка завершена» | оператор |
| `loaded → shipped` | ввод веса выезда + кнопка «Отгрузить» | оператор |

**Ключевое изменение vs текущей логики:** раньше CV `complete` сам дёргал
`record_loading` и переводил `arrived→loading`. Теперь `arrived→loading` — это
момент **загрузки видео**; CV `complete` только записывает мешки; `loading→loaded`
— ручная кнопка оператора.

## Бэкенд

### Order (`orders/models.py`)
- Добавить `"loaded"` в `STATUSES`, порядок: `… arrived, loading, loaded, shipped`.
- Миграция на новый статус.

### Сервисы (`shipments/services.py`)
- **`record_arrival(order, weigh_in_kg, user)`** — убрать параметр `truck_number`
  (номер уже в заказе). Guard `confirmed`/`paid` → `arrived`; пишет `weigh_in_kg`
  и `arrived_at`. Вызывается вебхуком entry и ручным fallback.
- **`start_loading(order, user)`** (новая) — guard `arrived` → `loading`.
  Вызывается при загрузке видео (момент «началась обработка»).
- **`record_count(order, bags)`** (переименование текущей `record_loading`) —
  пишет `bags_loaded` в shipment, **статус не трогает** (остаётся `loading`).
  Вызывается CV `complete`.
- **`finish_loading(order, user)`** (новая) — guard `loading` → `loaded`.
  Кнопка «Загрузка завершена».
- **`record_shipment(order, weigh_out_kg, user)`** — guard меняется
  `loading` → **`loaded`**. Считает `net = |выезд − въезд|`, списывает склад,
  `→ shipped`. В eventlog кладёт поля сравнения: `bag_estimate_kg = bags × вес_мешка`,
  `discrepancy_kg = net − bag_estimate_kg`. Возвращает эти поля в ответе.

### Эндпоинты (`shipments/`)
- `POST /orders/<id>/arrive/` — вес въезда (perm `shipping.arrive`).
- `POST /orders/<id>/load/` — оставить как fallback ручного ввода мешков без видео
  (perm `shipping.load`). Теперь зовёт `start_loading` + `record_count`.
- `POST /orders/<id>/finish-loading/` (новый) — кнопка «Загрузка завершена»
  (perm `shipping.load`), зовёт `finish_loading`.
- `POST /orders/<id>/ship/` — вес выезда (perm `shipping.ship`).

### Загрузка видео
- `UploadVideoView` при создании задачи дополнительно вызывает `start_loading`
  (заказ `arrived → loading`).
- `VideoCompleteView` зовёт **`record_count`** (не `record_loading`), статус
  остаётся `loading`. Заказ в `loaded` переводит только кнопка оператора.

### Вебхуки (`webhooks/`)
Маппинг сохраняется: `entry → record_arrival`, `counter → record_count`,
`exit → record_shipment`. Поиск заказа по номеру не меняется.

## Воркер и реал-тайм

CV-логику **не трогаем**. Воркер: `next/` → скачать видео →
`BagColorCounter.run()` → `+1` в Redis на каждый мешок → `complete{bags:N}`.
`complete` теперь зовёт `record_count`.

Реал-тайм под Mac не оптимизируем. Условие приёмки — корректная сборка и работа
через `docker compose up --build`; воркер запускается отдельно на GPU-машине по
README (команда с ключом counter-камеры, `CV_DIR`, `CV_DEVICE`).

## Фронтенд (`shipping/page.tsx`)

Стэппер: **Оплачен → Прибытие → Загрузка → Выезд** (4 шага; `loaded` входит в шаг
«Загрузка» как «готов к выезду»).

По статусам:
- **arrived (Прибытие):** view-only — номер, вес въезда, статус оплаты. Fallback
  поле «вес въезда», если вебхук не прислал. Кнопка «Начать загрузку» через
  загрузку видео.
- **loading (Загрузка):** «Загрузить видео» → живой счётчик мешков. После
  CV-complete — финальное число + кнопка **«Загрузка завершена»** (`→ loaded`).
- **loaded:** блок «Выезд» — поле веса выезда + предпросчёт сравнения + кнопка
  **«Отгрузить»**.
- **shipped:** итог сравнения (заказ-вес / груз-вес / расхождение), бейдж
  «Отгружено».

**Блок сравнения:** «Вес груза (выезд − въезд): X кг · Ожидалось (N мешков ×
вес): Y кг · Расхождение: ±Z кг». Красный при превышении порога, но отгрузку **не
блокирует** (предупреждение).

Оплата — базовая (показываем оплачен/нет; ворота позже; не блокируем).

## Сборка из коробки
`docker compose up --build` поднимает db + redis + backend + frontend; миграции
применяются автоматически (статус `loaded`). Воркер — вне Docker (CV-зависимости
~3 ГБ), запускается по README.

## Тестирование
- Сервисы: guard'ы каждого перехода (arrival, start_loading, record_count,
  finish_loading, record_shipment) — допустимые и недопустимые исходные статусы.
- Сравнение веса: `net`, `bag_estimate_kg`, `discrepancy_kg` считаются верно.
- Эндпоинты: `finish-loading/` (perm, переход), `ship/` с весом и сравнением.
- Видео: upload переводит в `loading`; complete пишет мешки, статус остаётся
  `loading`.
- Существующие 101 тест должны остаться зелёными (с учётом переименований).

## Вне scope (YAGNI)
- Реальные датчики веса (только вебхук-имитация + ручной fallback).
- Жёсткие ворота по оплате (будут позже).
- Ручная привязка нераспознанного номера.
- Оптимизация CV под CPU.
