# Полный CRUD на фронте для справочников

**Дата:** 2026-06-23
**Статус:** утверждён к реализации
**База:** main

## Цель

Сейчас на фронте только создание. Добавить **редактирование** и **удаление**
для справочников: Клиенты, Товары, Сотрудники, плюс удаление Камер. Бэкенд уже
поддерживает полный CRUD (ModelViewSet, PATCH/DELETE, права заведены) — работаем
только на фронте.

## Принципы
- Редактирование — **та же форма-модалка**, предзаполненная, PATCH вместо POST
  (как уже сделано для Ролей).
- Удаление — через новый **ConfirmDialog** с подтверждением; ошибку бэкенда
  (PROTECT / системная запись) показываем в диалоге, не закрывая его.
- Активность (toggle «Скрыть/Включить») остаётся как мягкое скрытие; жёсткое
  удаление — дополнительно.
- Кнопки скрыты без соответствующего права (`can(me, …)`).
- Бэкенд не трогаем (CRUD и права уже есть).

## Секция 1: ConfirmDialog (`ui/confirm-dialog.tsx`, новый)

```
ConfirmDialog({
  open, onClose,
  title,                       // «Удалить клиента?»
  description?,                // «Это действие необратимо.»
  confirmLabel = "Удалить",
  busy?, error?,               // busy блокирует кнопку; error — текст под телом
  onConfirm,                   // () => Promise<void>; бросает → error остаётся открытым
})
```
- На базе `Modal` (eyebrow «Подтверждение», title, footer: «Отмена» + кнопка
  `variant="destructive"` с `confirmLabel`).
- Тело: `description` + (если есть) блок ошибки.
- При неудаче `onConfirm` диалог НЕ закрывается; ошибка показывается.

## Секция 2: Редактирование (та же форма + PATCH)

Общий паттерн: форма получает опциональный `editing`-объект → предзаполнение +
PATCH `/{entity}/{id}/`; иначе POST. Заголовок/eyebrow модалки меняются на
«Изменить …» / «… · Изменение». Кнопка «Изменить» в строке открывает модалку с
`editing`.

- **Клиенты** (`clients/page.tsx`, `ClientForm`): добавить проп `editing?: Client`;
  `defaultValues` из него; в `onSubmit` — `editing ? api.patch(\`/clients/${id}/\`) :
  api.post`. Кнопка «Изменить» (perm `clients.edit`).
- **Товары** (`catalog/products/page.tsx`): `editing`-стейт (Product|null); при
  «Изменить» заполнить `name/color/weight/price`; submit PATCH. Inline-edit цены и
  toggle активности остаются. Кнопка «Изменить» (perm `catalog.edit`).
- **Сотрудники** (`management/employees/page.tsx`): `editing`-стейт; предзаполнить
  поля (кроме пароля); при edit **пароль необязателен** — если пустой, не слать в
  PATCH. Submit PATCH. Кнопка «Изменить» (perm `employees.manage`).

## Секция 3: Удаление + кнопки в строках

В колонке действий каждой строки: «Изменить» (Button outline) + «Удалить»
(Button ghost, иконка Trash2, красный hover) → `ConfirmDialog` → `api.delete`.

| Сущность | Изменить (perm) | Удалить (perm) | DELETE |
|---|---|---|---|
| Клиенты | `clients.edit` | `clients.delete` | `/clients/{id}/` |
| Товары | `catalog.edit` | `catalog.delete` | `/products/{id}/` |
| Сотрудники | `employees.manage` | `employees.manage` | `/employees/{id}/` |
| Камеры | — | `cameras.manage` | `/cameras/{id}/` (кнопка на детальной странице) |

- Кнопки скрываются по `can(me, …)`.
- **Камеры:** только удаление, на детальной странице `management/cameras/[id]`
  (кнопка «Удалить камеру» → ConfirmDialog → delete → редирект на список). Полный
  edit камеры — вне scope (частичный edit шаблона уже есть).
- При ошибке удаления (напр. товар используется в заказах — PROTECT) бэкенд
  возвращает ошибку; показываем её текст в ConfirmDialog.

## Тестирование

- `npm run build` — без ошибок после каждой сущности.
- Визуальная проверка в Docker: создать/изменить/удалить клиента, товар,
  сотрудника; удалить камеру; проверить, что удаление защищённой записи
  показывает ошибку в диалоге; кнопки скрыты у пользователя без прав. Light + dark.
- Юнит-тестов на верстку нет; бэкенд-suite не затрагивается.

## Вне scope (YAGNI)

- Изменение остатков напрямую (Склад — движения через adjust).
- Полное редактирование Камер (только удаление + существующий edit шаблона).
- Детальные страницы `[id]` для справочников (edit — в модалке).
- Изменения бэкенда.
